# Code Security Rules (MANDATORY)

- Never trust user input — always validate and sanitize.
- Use allowlists over denylists for input validation.
- Prevent SQL injection by using parameterized queries (no string concatenation).
- Avoid command injection — never pass raw input to shell/system calls.
- Escape/encode output to prevent XSS (HTML, JS, URL contexts).
- Use secure authentication practices (no plaintext passwords).
- Always hash passwords with strong algorithms (bcrypt, argon2).
- Never hardcode secrets (API keys, tokens, credentials).
- Use environment variables or secure vaults for secrets.
- Apply the principle of least privilege (minimal permissions everywhere).
- Validate file uploads (type, size, content) and never trust filenames.
- Prevent path traversal (sanitize file paths, restrict directories).
- Use HTTPS everywhere — never send sensitive data over HTTP.
- Protect against CSRF (tokens, same-site cookies).
- Set secure headers (CSP, HSTS, X-Frame-Options, etc.).
- Avoid exposing internal errors or stack traces to users.
- Log security-relevant events, but never log sensitive data.
- Rate-limit critical endpoints (auth, APIs) to prevent abuse.
- Use timeouts and limits to prevent DoS attacks.
- Regularly update dependencies to patch vulnerabilities.
- Avoid unsafe deserialization of untrusted data.
- Validate and constrain JSON/XML inputs (prevent XXE, bombs).
- Use proper session management (secure cookies, expiration, rotation).
- Invalidate sessions on logout and sensitive actions.
- Avoid insecure randomness — use cryptographically secure generators.
- Review third-party code and dependencies before using them.
- Separate user data and enforce proper access control checks.
- Fail securely (deny by default on errors).
